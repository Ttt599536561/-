## 4. 当前问题清单

### P0 问题

- 远端课表同步仅完成第一版，仍未形成完整双端一致性
  - 当前更多是“页面进入 / 保存动作触发同步”
  - 还不是完整实时或事务型同步
  - 导入替换仍缺少服务端事务保护

- 远端专属约会主链路虽然已接通，但仍未完成稳定验收
  - 仍缺少带鉴权双机实测结论
  - 同步失败后的自动恢复 / 重试策略不足
  - 共同空闲块创建约会时仍缺少冲突校验

- 共享待办仍是本地 Room 闭环
- 快捷提醒仍只是本地提示文案，没有真实发送与历史记录
- 纪念日基础字段已远端同步，但互动能力还未延展成提醒闭环

- 发布安全配置仍未达标
  - 签名敏感信息仍在工程内
  - 明文流量和 SSL 策略仍需收敛
  - release 网络日志仍需严格区分

- 导入稳定性仍需继续提升
  - 失败诊断
  - 错误文案
  - 适配兼容性
  - 成功率

### P1 问题

- 共同页当前只完成第一版
  - 专属约会详情 / 回复已完成，但冲突校验未做
  - 更丰富的推荐文案未做
  - 是否需要多轮消息 / 回复历史仍未产品定版

- 空档页 `GapFinderScreen` 与新共同页的职责边界仍需继续梳理

- 埋点当前只是本地骨架，未接入真实分析平台

- 系统提示还未做到全项目绝对统一

- `RegisterScreen` 已经不是核心入口，仍有简化空间

### P2 问题

- 多个 Screen 文件仍偏大
- 缺少自动化测试
- 当前中文路径下已可构建，但仍依赖 `android.overridePathCheck=true`
- README / ARCHITECTURE 还有少量描述未完全跟上当前状态

## 7. 需求清单

### 7.1 必要需求

1. 用户必须可以真实登录，并在不同设备上恢复自己的身份和数据。
2. 用户必须可以真实绑定 TA，并在不同设备上读取同一份情侣关系。
3. 用户必须可以稳定导入、编辑和删除自己的课表。
4. 系统必须可以基于双方课表计算共同空闲时间。
5. `共同` Tab 必须以“共同空闲主视图”为核心。
6. 用户点击共同空闲块后，必须可以创建专属约会。

### 7.2 重要需求

1. 共同页必须优先保证性能和跟手性，再追求高级视觉。
2. 共同页状态必须清晰区分：
   - 共同空闲
   - 我有课
   - TA有课
   - 都有课
   - 专属约会
3. 任一方课表未准备好时，系统不得误判为共同空闲。
4. 所有系统提示必须符合人机交互学原则。
5. 专属约会必须支持查看详情、标题锁定后的回复，并保持双端一致。
   - 当前实现是“对方单次回复”，不是多轮聊天
6. 课表同步第一版必须确保：
   - 登录后可拉远端课表
   - 编辑 / 删除 / 导入会同步远端
   - TA 课表可以从远端拉取

### 7.3 发布前需求

1. release 安全配置必须达标。
2. 远端课表同步必须稳定。
3. 共同页和专属约会必须具备双端一致性。
4. 至少具备基础单元测试、lint 和构建验证流程。
